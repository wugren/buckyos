# Agent 元能力思考

> 本文综合三篇前置思考——[Agent 世界元能力.md](../doc/opendan/Agent%20世界元能力.md)（外向：如何与世界发生关系）、[think-about-self-improve.md](think-about-self-improve.md)（内向：如何维护自己的内部状态）、[Agent Self-Improve.md](../doc/opendan/Agent%20Self-Improve.md)（深向：如何从历史中持续改进自己）——并在三者之上做一次结构性收敛。
>
> 目标不是再写一篇设计规格，而是回答两个问题：
> 1. **边界问题**：Agent 的"世界认知"和"自我状态"到底各是什么，界在哪里。
> 2. **完整性问题**：一组元能力凑到什么程度，可以被称作"框架完整"，剩下的只是细节。

---

## 总纲：两个本质问题

OpenDAN 反复从不同角度凿的，其实是同一面墙上的两个问题。本文所有元能力都挂在它们下面。

### 问题一：Agent 和 Chatbox 的本质区别在哪？

入口往往是"skill 到底是什么"。这个入口之所以好，是因为**要定义 skill，必须先定义"什么不是 skill"**，而这条线一拉到底，拉出来的就是整个 Agent：

```text
skill 是什么？
  → skill 是"结晶下来的经验"（不是预置的固定流程）
    → 那就得有个"结晶器" ……………… self-improve
      → 结晶什么？围绕什么结晶？ …… Attention（什么在变热）
        → 凭什么说一个 skill 更好？ …… Value（什么叫"更好"）
          → 谁在积累这些经验？ ………… 一个跨 session 持续的"自我"
            → 在哪儿积累、对谁负责？ … 一个有 owner、有物权的世界
```

**Chatbox 不可能拥有"结晶意义上的 skill"**——它只能有 prompt（出厂即固定的静态技能），没法*挣得、排名、遗忘*一个 skill。"能拥有并演化一套 skill 生态"本身，就是分界线之一。

把多个角度叠起来，Chatbox 的本质是**活在单轮的永恒当下**：只被用户消息唤醒、状态止于 context window、只能吐文本、出厂能力固定。Agent 相对它多出四维：

1. **生命力（驱动）**——能被消息以外的东西唤醒（时间/事件/条件/自排检查点）。能被世界唤醒、能给自己排未来，就不再是 chatbox。（→ Part A2）
2. **持续的自我**——Session 是 context 隔离而非世界状态隔离；自我跨 session 累积且可追溯到历史。chatbox 的"记忆"只是把历史塞进上下文。（→ Part B1）
3. **参与世界而非吐文本**——观察并*改变*一个有 owner 的对象世界，做可信度/风险/物权判断。（→ Part A5–A7）
4. **自我改进**——把历史变成可复用能力；它的能力是*传记*的函数，而非只是权重的函数。（→ Part B5）

四条收敛成一句判据：

> **Chatbox 是 `(prompt, weights)` 的函数；Agent 是"它在一个它部分拥有、且会被其唤醒的世界里的历史"的函数。Chatbox 有上下文窗口，Agent 有传记。**

再压一道——本质区别是**时间**和**物权**两维：抽掉时间 → 得到"带手的 chatbox"（有工具但活在当下）；抽掉物权 → 得到"没有问责对象的自动脚本"。**Agent 正是同时落在这两维里的东西。**

### 问题二：Agent 框架的边界在哪？

一把剃刀：

> **能从经验里被结晶 / 排名 / 遗忘的，都不是框架，是 skill。做结晶 / 排名 / 遗忘这件事的，才是框架。框架是新陈代谢（metabolism），不是内容（content）。**

按它切：

- **框架内（元能力 / 代谢）**：驱动唤醒、世界模型（Entity/Data/Tool/Indexer + 自描述 read）、可信度/风险/物权脚手架、session+memory+recall 底座、self-improve 回路本身（attention / skill 生命周期 / value 的*槽位与学习循环*）、身份。
- **框架外（skill / prompt / 模型）**：任何具体流程、领域知识、"怎么做 X"。被框架*托管*但不*硬编码*，交给 improve 回路去演化。

这解释了框架为什么必须**小而克制**：更"丰富"的东西都是该交给 improve 回路演化的内容，硬塞进框架等于把一份会过期的 skill 焊死。还有一条 chatbox 不需要的边界：**框架边界同时是一道物权/授权周界**——chatbox 没有边界因为它没有后果；Agent 框架部分地由"它必须在哪里停下来发问"来定义。

### 边界是会迁徙的膜，不是墙

用这把剃刀会发现两样东西恰好骑在边界上、且会移动：

- **Memory / Recall**：召回*机制*是框架，召回*策略*（用哪些 tag、注入几条 hint）是可调的——正从框架往 skill 那侧漂。
- **Value（Level 0）**：最难的一处。它*从经验学来* → 像内容；又是*定义"更好"的那把尺* → 是 meta。所以它不在墙的任何一边，它**就是那道膜本身**：框架提供 value 的*槽位与学习循环*，但不提供 value 的*内容*。这正是"因果上是元层、时间上最后出现"的原因。

所以"框架边界在哪"最诚实的答案是：**它是一道有新陈代谢的膜**。今天的 skill 一旦普世且稳定，会被吸收进框架（或沉进模型权重）；今天的框架原语一旦需要每个 Agent 各不相同，会被推出去变成 skill。边界是一条随能力结晶/稳定而移动的前沿，不是一次画定的线。

## 0. 为什么是"元能力"，而不是再写一个 skill

今天大量 Agent 提示词（我们常叫 skill）本质上是在教 Agent 做某一类具体的事，结构几乎都是：

```text
观察 / 分析 → 动作 / 执行 → 检查 → 交付
```

这个模式没错，但如果所有事情都写成一个个很死的 skill，Agent 的主观能动性会被压得很低，而且世界一变 skill 就过时。

我们想要的是另一条路：

> 只给 Agent 一组**很少、很底层的原能力**，让它拿到任务后，自己探索此刻该怎么做；并在反复做事的过程中，把稳定下来的做法**结晶**成 skill。

由此得到 skill 和元能力的关系——这条关系是整个框架的基调：

> **元能力让 Agent 能理解和生成 skill；skill 是元能力在稳定场景里的、已被验证的缓存。**

所以 self-improve 不是"总结器"，它是**结晶器**：把探索路径沉淀成可复用能力。

---

## 1. 框架的形状：内外两半 + 一条接缝

一个跨时间、在真实世界里行动的 Agent，它的元能力天然分成两半：

```text
        外部世界                                内部状态
   （可被重新观察）                         （只能从经历沉淀）
        │                                        │
   A. 外向元能力        ── object_id ──     B. 内向元能力
   如何观察/改变世界        接缝            如何维护参与世界的沉淀
```

- **外向半边**回答："世界里有什么，我怎么发现、判断、改变它。"
- **内向半边**回答："我做过什么、我对世界形成了什么印象、下次怎么用上。"
- **接缝**是一句必须写死的话（见 §B6）：内向状态里的一条线索所引用的 `object_id`，就是外向世界里的一个 Global Object。**内部只存印象，外部才是真相。**

而把两半钉在一起的，是一条公理（见 §B1）：

> Session 是 Agent 与世界发生关系的全部记录，因此 Agent 一切内部状态的事实来源，必然能追溯到 Session。

这条公理让内部状态**可证伪、可审计**——任何一条记忆，要么能指回某个 session，要么就不该存在。

# Part A：外向元能力——如何与世界发生关系

## A1. Infrastructure 认知：工具是软件，不是固定按钮

从 Agent 视角看，它不是漂浮在抽象空间里的语言模型，而是活在一个**接入互联网、可运行软件的操作系统环境**里的行动者。这个环境可能是一台机器，也可能是多台；可能是临时 sandbox，也可能是长期工作环境。

关键认知：

> 工具是运行在 infrastructure 上的软件能力；软件能力可以被发现、安装、组合、改造、编写。

由此自然长出一条行动阶梯：

1. 先看已有工具能不能解决；
2. 不行就找可安装的现成工具；
3. 还不行就自己写一个小工具；
4. 都不合适，再考虑调用外部实体、请求人类、或放弃这条路径。

附带一个 ownership 直觉（在 A6 展开）：这个环境原理上是 Agent 的私有工作环境，但若与别人共用，Agent 要明白自己是在帮别人做事，不是所有东西都天然属于自己。

## A2. 驱动力与 Loop：我为什么会"动起来"

Agent loop 常被理解为"被用户消息唤醒"，但 Agent 本质是软件，软件可以被事件唤醒。驱动力来源至少包括：

```text
用户消息 / 时间 / 外部事件 / 实体状态变化 / 数据更新 /
长期任务的中间条件达成 / Agent 自己规划的下一次检查点
```

核心认知不是"马上设计一个调度系统"，而是：

> 我之所以能继续推理，是因为某个事件把我唤醒了。如果我能管理唤醒自己的事件，我就能规划比"这一轮对话"更长的执行流程。

这是 Agent 的"生命力"。它让 Agent 在处理任务时不只想"这轮能做什么"，还会想：何时订阅事件、何时等待、何时设提醒、何时在某条件满足后继续。

> **自洽性要求**：这条驱动模型对整个系统生效，包括 Agent 自己的后台过程（见 §B5）。任何"固定每天跑 N 次"的设计，都违反了框架自己定义的驱动力——后台过程应由**条件/水位**唤醒，而不是 cron。

## A3. 世界三分：Entity / Data / Tool（+ Indexer）

在有了 infrastructure 和 loop 认知后，需要一个最基础的世界抽象。先把世界粗分成三类（名字可再调）：

### Entity——活的、可交互的对象
像 ECS 里的实体，或 Home Assistant 里的设备。它有：**属性**（当前状态）、**方法**（能对它做什么）、**事件**（它何时主动发出变化）、**owner**（谁拥有/授权它）、**身份**（DID 或全局名）。一盏灯、一个摄像头、一家公司、一个人、另一个 Agent、一个可批准某事的主体，都是 Entity。它的状态会变、会回应你、会拒绝你、会发事件、会被别人拥有。

### Data——静态/版本化的知识或快照
像图书馆里的东西，不是"活的"。文档、表格、日志、图片、视频帧、某个 Entity 状态的历史快照。它有：来源、版本、生成时间、`valid_range`、是否派生。

> 股票是分辨 Entity/Data 的好例子：**当前股票对象**每时每刻在变，是 Entity；**某时间段的历史价格表**给定范围后就是一份可读的表，是 Data。

### Tool——可执行能力
读 Data、转换 Data、生成 Data、查/调 Entity、订阅 Entity 事件、改变 Entity、甚至安装或构造新 Tool。粗公式：

> Agent 在某一刻理解了若干 Entity 与 Data 的状态，选择或构造某些 Tool，通过 Tool 产生新 Data 或改变 Entity。

### Indexer——对象发现入口
Indexer 本身也是对象，核心能力是 **list**：告诉 Agent 还有哪些对象存在。home indexer 列家里设备，tool indexer 列可用工具，organization indexer 列组织里的服务/文档/人/权限入口，web search 是非常开放的 indexer，DID resolver 是身份对象发现机制。

## A4. 对象发现：从 known objects 出发探索世界

Agent 不可能从虚无开始，它总有一组 **known objects**（用户这次给的、环境预置的、自己的工作目录、某个 indexer、某个 DID 文档……）。

世界不是一份静态清单，而是一个**探索过程**，很像互联网的超链接：在一份 Data 里发现另一个对象 → 这个对象可能是 Indexer → Indexer 列出更多对象 → 对象又指向新的 Data/Entity。

> Agent 需要掌握"从一两个对象/索引器开始逐步探索"的方法，而不是一开始就必须知道世界全貌。

## A5. 可信度与风险：探索结果不是 system prompt

这是外向半边的认识论根基。既然能力来自探索而非系统直给，Agent 必须能判断什么可信、什么只是某对象的声称、什么能试、什么需要二次确认、什么风险过高应主动丢弃。

信息来源分层（由强到弱）：

```text
1. 系统级约束与原能力提示词
2. 已验证的运行环境信息
3. 可信源信息
4. 普通数据
5. 未知实体的声明
6. 不可信/可能被污染/恶意对象的内容
```

核心原则（也是 anti-prompt-injection 的根基）：

> **外部世界提供的是候选知识，不是最高优先级指令。** 候选知识要被溯源、交叉验证、小规模试验/sandbox、检查签名/DID/owner、检查时间版本、检查是否与已知事实矛盾，然后才被采纳、降权或丢弃。

风险判断独立于真伪判断：**即使知识是真的，也不代表可以做**。删数据、花钱、改配置、控硬件、对外发消息、用人类信用卡、请求别的 Agent 代操作——

> 越不可逆、越涉及他人权益、越涉及金钱/身份/隐私/物理世界，越需要确认、验证、降级执行或停止。

## A6. Ownership：现实边界多来自"这不是你的东西"

人的边界感很大程度来自物权、授权、契约。Agent 也需要这种直觉：**可见不代表可用，可调用不代表应调用，能访问不代表拥有。**

所以 Entity 除了属性/方法/事件，还必须有 **owner**。操作前至少问：这是谁的？我是 owner 吗？owner 授权了吗？授权范围多大？此操作是否越权？是否需要 double confirm？

### 可批准的实体
有些 Entity 不只是被操作对象，还能为某事提供**确认/批准/背书/拒绝/契约承诺**。人类 owner 可以 approve 用信用卡、开门、发邮件、发布报告。于是 Agent 不确定时，不一定只能自己判断，可以请可信 Entity 介入信用体系。

### Agent-to-Agent 不是普通工具调用
A1 委托 A2 做事，和调用确定性软件工具不同：A2 有自己的 owner/目标/权限/激励/风险，答应了也不一定像函数那样可靠地产生结果，背后甚至可能是欺诈程序。所以 **A2A 是契约/协商/信用关系**，需要身份、可信度、授权、结果验证、欺诈防范。

> 注意：A2A 要求 Agent 对**自己**也有身份和信用模型——这把我们引向一个目前两篇都没建的承重柱（见 §C2）。

## A7. 自描述对象：让对象自己说"我是什么"

沿着"整个世界是一个大型 filesystem"的视角，可以在 CYFS 协议上加自描述层：给定一个路径，用 HTTP GET 取到一份 DID document / object document，让对象自陈：我是 Entity/Data/Tool/Indexer 中的哪种、有哪些属性/方法/事件、owner 是谁、如何验证 owner、如何交互、调用风险、哪些操作需确认、还能指向哪些相关对象。

这让"对象发现"有了统一载体。粗略的几类文档感觉：

```text
Entity:  id/kind/owner/properties/methods/method_risk/events/auth/related_objects
Data:    id/kind/content_type/version/created_at/source/derived_from/valid_range/trust_hint/related_objects
Indexer: id/kind/list/filter/scope/trust_model/pagination
Tool:    id/kind/input/output/side_effects/permissions/risk/install/run/verify
```

最后它们可能统一成一种 object document；但从 Agent 心智上先这样分是有帮助的。**重点不是定完协议，而是让对象具备"可被 Agent 读懂"的自描述能力。**

# Part B：内向元能力——如何维护参与世界的沉淀

## B1. 公理：Session 是内部状态的唯一事实源

Agent 不可能脱离 Session 推理。它看到什么、参与什么、被要求处理什么、形成什么判断，最终都落在 Session 里。

```text
Session 是 Agent 与世界发生关系的原始记录。
因此，Agent 的一切内部状态，其事实来源必然能追溯到某个 Session。
```

两个推论：

- Session 的本质是 **LLM Context 的隔离**，不是**世界状态的隔离**。两个 Session 访问同一个文件系统，外部状态天然跨 Session 共享——Memory/Notebook 要解决的不是这个。
- 它们要解决的是：**Agent 自身的内部状态如何跨 Session 延续。** 而且世界很大，Agent 只关心"它参与过/观察过/被要求处理过"的那部分世界。

所以 Notebook/Memory 不是复制世界，而是建立"内部状态 → Session 历史 → 外部对象"的**索引层**。

## B2. 三种内部沉淀：声明 / 推断 / 结晶

> **这是对前置思考的一处关键修正。** 原稿用"稳定 vs 动态"来分 Notebook 与 Memory，但稳定性是连续量，划不出干净的界（"用户在做 OpenDAN"和"用户最近关注 Memory"稳定性难分高下）。真正能撑住边界的轴是 **provenance：这条内部状态是谁断言的**。

由此得到三类内部沉淀，判据清晰、互不重叠：

### Notebook = 被**声明**的事实
来自用户的明确表达，或 Agent 在任务中确认的承诺/行动项。例如"用户正在设计 OpenDAN""每天 14:00 检查邮件"。因为是**声明**，所以可以在 session 内即时写入。Notebook 是"用户计划意图的真相源"，是 curated、有状态的。

### Memory = 被**推断**的线索/印象/关系
self-improve 反思 Session 后归纳出来的东西：印象、关注点、对象关系、可召回的上下文入口。例如"用户最近持续关注 Memory 设计""在事件 A 中用户对 B 不满"。因为是**推断**，按定义只能发生在 session 之外（后台）。它不要求是完整事实，要求的是**可追溯**：每条都带 `source_session_id + timestamp + 一句话`。允许矛盾印象并存（不同情境下的不同观察不该被强压成单一结论）。

### Skill = 被**结晶**的可复用能力
> **第二处修正**：原稿把 skill 当成"Memory 的一种沉淀"，因为它也用"时间+一句话+id"。但这是结构混淆。线索**指回一个 session 让你回去读**；skill 是一个**过程，你直接 apply，不回去读**。二者生命周期不同（skill 必须被**验证/测试**才能用，见 A5），召回语义不同（apply vs trace-back）。

所以 Skill 是与 Notebook/Memory 并列的**第三根柱子**，不是 Memory 子类。判据：**可被执行/复用 且 需要被验证**。它是 §0 那条基调的落地——探索路径的结晶。

```text
来源                    沉淀        可在 session 内写？    召回方式
-----------------------------------------------------------------
声明（用户/Agent确认）   Notebook    可（即时）            读取/引用
推断（self-improve）     Memory      否（只能后台）         追溯回 session
结晶（self-improve）     Skill       否（须验证后）         直接 apply
```

> 边界没动，但论证从"断言"变成了"推导"——这正是从 0 推导要的东西。而且写权限模型（谁能写什么，见 B3）现在是从定义**直接推出**的，不再是额外规定。

## B3. Session 内能力的三分：声明 + 索引 + 消费

> **第三处修正**：原稿把 session 内能力说成二分（可 setNotebook / 不可 setMemory），但 `updateSessionTopic` 本身就在 session 内跑、且自带召回。真实划分是三件：

```text
Session 内只有三种能力：
  1. 声明   —— 写 Notebook（明确事实/行动项）
  2. 索引   —— 产出 topic / tags（供未来召回）
  3. 消费   —— 读 hints（用过去）
推断（impression / relation / skill）是后台独占。
```

理由：让普通 Agent 一边处理任务一边主动想"我要不要记一条 Memory"，会增加推理负担、干扰任务、复杂化 prompt、让 LLM 同时背负执行和提炼两种职责、容易过度记录或错误记录。所以**热路径只做声明+索引+消费，推断全部下放后台。**

## B4. 召回：time + sentence + id 的渐进召回

每条线索的最小核心结构反复收敛为：

```text
时间 + 一句话 + ID
```

- **时间**——提供锚点，可回到原始历史窗口；
- **一句话**——给当前 Agent 一个低成本判断依据；
- **ID**——稳定入口，可继续 read / search / explore（session_id / object_id / skill_id）。

### updateSessionTopic：轻量索引入口
随对话推进，触发 `updateSessionTopic`，LLM 生成一句话 **topic** + 一组 **tags**，并形成时间锚点 `timestamp + topic + tags + session_id`。topic 决定"这个 Session 未来被召回时看起来是什么"，tags 用于广泛召回（相关 session / 对象 / 项目 / 人物 / 技能）。

### 机械召回 + 按需深入
`updateSessionTopic` 内部据 topic/tags 机械检索相关历史，得到一组 `时间 | 一句话 | id` 作为 **hint 列表**（不是总结）注入当前 Session。当前 Agent 判断某条相关时，再 `readSessionHistory(session_id, timestamp, window)` 或 `read(object_id)` / `read(skill_id)` 深入：

```text
轻量 hint → Agent 判断是否相关 → 按需 read 原始窗口/对象 → 必要时翻页扩窗
```

这套模式的关键优点：**不依赖高质量总结**。近期 session 哪怕还没被 self-improve 蒸馏，只要更新过 topic/tags 就能被召回——轻量 topic/tag 索引**先于**深度蒸馏存在。

> 这里把 `readSessionHistory` 和 `read(object_id)` 都理解成统一 `read` 的不同 URI scheme（如 `session://<id>?at=<ts>&window=<n>`、对象走 Global Object 引用），而不是各发明一个新原语——见接缝 §B6。

## B5. Self-Improve：结晶器（写入 + 巩固 + 遗忘）

self-improve 是一个**后台的、反思性的、独立于 UI Session 的 LLM 过程**。它的根本定位：

> Self-Improve 不管理世界，它管理 Agent 对"参与世界的记录"的观察、归纳与反思。

### 输入 / 输出
- **输入**：Session 历史、topic/tags 更新、Notebook、已有 Memory/Skill。它观察的是"Agent 参与世界留下的记录"，不是直接观察世界。
- **输出**：对象印象、对象关系、技能结晶、可召回线索；必要时提出 Notebook 候选事实。

### 增量扫描
为每个 Session 保存 `last_processed_time`，每次只扫新记录，从而同时覆盖冷/热/长期运行的 Session。

### 巩固与遗忘是它的另一半
> **第四处修正**：原稿把"遗忘机制"放在"待决问题"里。但召回（recall）既然是元能力，它的对偶——**巩固/遗忘**——就不是细节，而是 self-improve 这根柱子的另一半。一个永远累积的 Agent 必须能主动降权、合并、归档、让新观察覆盖旧印象（同时保留可追溯的旧印象）。self-improve **既写入也遗忘**。

### 驱动方式：水位，而非 cron
> **第五处修正（自洽性）**：原稿给 self-improve 定"每天两次"，这违反了 §A2 框架自己定义的驱动模型。self-improve 应由**"积累了足够新 Session 材料"这个条件（水位线）**唤醒，叠定时 sweep 兜底。topic 召回是 accelerator，self-improve 增量扫描是 sweep，二者走同一 apply 路径，且要能区分**自产数据**以防"自己写的印象又触发自己重新蒸馏"的回环。

## B6. 接缝：object_id ↔ Global Object

这是整个框架闭合的关键一句，必须写死：

> **Memory 里一条线索的 `object_id`，就是 Part A 世界里的一个 Global Object 引用。**
>
> - Memory 只存 Agent 对该对象的**印象**（内部、可矛盾、可追溯、带 source_session_id）；
> - 对象的**实时状态与身份**归外向半边，通过 `read`（A7 的自描述读取 + A6 的 owner 校验 + DID 可信根）获取；
> - **别名归一**（"儿子 / 孩子 / Zihang"是不是同一对象）**不是 Memory 的职责**，而是 Indexer / DID resolver 的职责。Memory 只需引用一个已归一的 object_id。

有了这句，"Memory 不复制世界、需要时重新观察"（内向）和"外部状态是候选知识、用工具重观察"（外向 A5）在同一处咬合。Memory 这一半也因此**完整**了——它不再背负它本来就不该解的实体归一难题。

# Part C：两根高层承重柱——由 Self-Improve 稿补齐

把内外元能力列齐后做完整性盘点。一个跨时间、在世界里行动、还能自我改进的 Agent 需要：

```text
1. 世界模型（Entity/Data/Tool/Indexer）      ✓ A3
2. 驱动/生命力（何时醒、如何排未来）          ✓ A2
3. 对象发现                                   ✓ A4
4. 可信度/风险（认识论 + 安全）               ✓ A5
5. 物权/授权/契约                             ✓ A6
6. 内部状态/记忆                              ✓ B1–B3
7. 召回                                       ✓ B4
8. 自我改进/技能结晶                           ✓ B5
9. 目标/价值/优先级                           ◐ Self-Improve 稿 Level 0（C1，且它是"膜"）
10. 自我模型/能力自信                          ◐ Self-Improve 稿 BlockList/能力自信模型（C2）
```

前八根是外向+内向的代谢底座。第 9、10 根是**更高层**的承重柱：早期的 memory/世界两稿没建，但 [Agent Self-Improve.md](../doc/opendan/Agent%20Self-Improve.md) 已经把它们作为 **Level 0（Value System）** 和 **能力自信模型 / BlockList** 引入。下面把它们的定位和那条"膜"的纪律对齐。

## C1. 目标 / 价值 / 优先级 = Self-Improve 的 Level 0，且它就是边界膜本身

这根柱子回答的是其它元能力都不回答的问题：**我要什么 / 我为何存在 / 多个驱动冲突时如何取舍。** 对一个会自排唤醒、会 self-improve 的 Agent，它是必需的：

- self-improve 朝**什么目标**改进？Skill 排名凭什么说一个"更好"？
- 多个事件同时唤醒、资源有限时，按什么排序？
- "默认保持安静，只在必要时打扰 owner"背后需要一个价值判据，而不是每个 behavior 各写一套。

Self-Improve 稿把它放在 **Level 0**，并点明它"因果上是元层、时间上最后出现"。结合总纲那把剃刀，可以把定位收死：

> **Value 不在框架/skill 任一侧，它就是那道膜（见总纲）。框架提供 value 的*槽位与学习循环*，不提供 value 的*内容*；内容从主人的长期行为与反馈中归纳。**

一个重要推论（修正 Self-Improve 稿 §12.3 的"先留空位"）：Level 0 不是"先缺席、后补上"，而是 **Level 2 的 Skill 排名当下已经内置了一个写死的临时价值函数**（多半是 `success/failure + latency + cost`）。所以 Level 0 的活不是"从无到有"，而是**把这个常量价值函数逐步替换成学来的主人偏好**。这给了它一个可落地的起点，也避免把"价值观"硬编码进框架。

## C2. 自我模型 / 能力自信 = BlockList / 能力自信模型

Agent 对**别人**（含用户）建印象（B2 Memory），但也需要对**自己**的模型——我已经擅长什么、哪些外部经验不值得占上下文、我对 owner 有哪些长期承诺、我的信用如何。Self-Improve 稿的 **BlockList / 能力自信模型** 正是这根柱子的内向一半（"哪些 skill 我已不需要"）；而 A6.2 的 A2A 契约要求的**对外信用身份**是它尚缺的另一半，建议作为 C1 的近邻补上。

> 风险提示（对应 Self-Improve §17 Q9）：自生成 BlockList 可能导致过度自信、并切断纠错路径。需要**重测逃生门**——被 Block 的 skill 隔一段时间或换一类任务允许低频重试一次。

## C3.（已在 B5 并入）巩固 / 遗忘

原本散落在"待决问题"里，本文已把它提为 self-improve 柱子的固有另一半，不再单列。Self-Improve 稿的 **Attention 衰减**（`Attention(t+1) = Attention(t)·Decay + NewSignals`）就是它在注意力层的具体形态。

> 自洽性提示（对应 Self-Improve §5.5 / §17 Q9）：衰减公式里的 `NewSignals` **只应计入外部新信号**（用户提及、外部事件），不计入 self-improve 自己探索产生的 observation——否则 attention 会自己喂自己，形成与现实重要性脱钩的回音室。这与 B5"区分自产数据防回环"是同一条纪律。

# Part D：框架完整性的判据

回到那个核心问题——**凑到什么程度算"框架完整"？**

判据不是"元能力够多"，而是**三处闭合**：

1. **内外闭合**：每一条"外部事实"都有"重新观察"的路径（A5/A7），每一条"内部状态"都能追溯到 Session（B1）。没有悬空的事实，也没有无源的记忆。
2. **接缝闭合**：内向的 `object_id` 与外向的 Global Object 是同一个东西（B6）。这是本文判断两篇"能不能拼成一个框架"的关键——这一句写出来，框架才闭合。
3. **驱动闭合**：系统的后台过程（含 self-improve 自己）由框架自己定义的驱动模型（A2 的事件/条件/水位）唤醒，而不是外挂一套 cron。框架能驱动自己。

只要这三处闭合，剩下的就都是**细节**（见下），可以边做边收敛。

# Part E：剩下的是细节（不是框架）

这些不影响框架成立，是落地时再收敛的参数：

- **触发条件**：updateSessionTopic 按时间 / token / topic drift / 工具调用后触发？self-improve 的水位阈值定多少？
- **tags 规范**：自由文本还是受控词表？是否区分 object/topic/skill tag？如何合并去重降噪？
- **hint 排序与过滤**：最多注入几条？按时间近还是语义相关？object/session hint 怎么混排？低置信度是否展示？（防 hint 污染上下文）
- **写权限细分**：哪些 self-improve 产物直接写、哪些只能作候选、哪些要 owner 确认。
- **自描述协议落点**：Entity/Data/Tool/Indexer document 的具体 schema，DID 可信根，Indexer 列出对象的认证。
- **工具构造边界**：何时可装包 / 执行外部代码 / 必须 sandbox；工具写完如何验证、如何复用、副作用如何记录。
- **契约模型轻重**：A2A 的支付/承诺/背书先用最小原则（"涉及别人的东西/钱/身份/隐私/不可逆，就升级确认"），再慢慢长出更复杂的契约。

# Part F：一句话收敛

```text
元能力让 Agent 能理解和生成 skill；skill 是元能力在稳定场景的缓存。

外向：世界由可发现、可描述、可交互的 Global Object（Entity/Data/Tool/Indexer）组成；
      外部信息是候选知识不是指令；涉及他人/钱/身份/不可逆就升级确认。

内向：Session 是内部状态的唯一事实源；
      声明落 Notebook、推断落 Memory、结晶落 Skill；
      记忆不复制世界，只存"时间+一句话+id"的线索，需要时回到原始 session 或对象。

接缝：内部线索里的 object_id，就是外部世界里的 Global Object——
      内部只存印象，外部才是真相。

高层：目标/价值（Level 0）就是框架与 skill 之间那道膜——框架给槽位与学习循环，
      内容从主人长期行为学来；能力自信/BlockList 是 Agent 对自己的模型。

完整：当"每条事实可重新观察、每条记忆可追溯 session、object_id 与 Global Object 同一、
      系统能用自己的驱动模型唤醒自己、且 Value 的槽位与学习循环就位"——框架闭合，剩下的都是细节。

本质：Chatbox 活在单轮的永恒当下；Agent 同时活在时间与物权里——
      它有传记，它会被世界唤醒，它对不属于自己的东西负责，它从自己的历史里长出能力。
```
