# BuckyOS 包管理系统：NodeDaemon 通过 PackageEnv 实现自动安装与升级流程

## 一、包加载（Load Package）的几种方式

### 1. 按包名加载（开发模式）

最常用的方式。使用者调用 `load_package` 时只传入 **package name**。

- NodeDaemon 通过 `load_package("app_name")` 获取该包的实际目录
- PackageEnv 负责返回实际目录路径
- NodeDaemon 拿到目录后，调用其中的启动脚本来启动 APP
- 这与开发模式下的目录结构天然吻合

### 2. 按包名 + 版本表达式加载

`load_package` 时传入 package name 及**版本运算符**，支持三种情况：

| 情况 | 说明 |
|------|------|
| 不带版本表达式 | 加载带有 `latest` 标签的版本；若无 `latest` 标签，则默认取版本号最大的 |
| 范围表达式 | 如 `>1.0, <2.0`，PackageEnv 支持类似 OS 包管理器的复杂版本运算 |
| 精确版本号 | 锁定到某个确定版本 |

> **备注：** PackageEnv 的设计早于 BuckyOS，最初是为了解决 Python 多依赖问题而设计的基础组件，因此天然支持复杂的版本运算。但对 BuckyOS 而言，**版本运算符能力基本用不上**——原因是我们的依赖关系 100% 不是文件级的，而是通过服务端点实现的。在分布式系统中，A 依赖 B 实际上是通过调度器获取服务 B 的端点后进行调用，各组件甚至不在同一个进程中。

### 3. 按包名 + ObjectID（哈希）精确加载

这是 BuckyOS 中**真正区分开发形态与运行形态**的关键：

- **开发形态：** 所有包未经发布，没有哈希，退回到按包名加载
- **运行形态：** 通过应用商店、系统更新等正规渠道获取的版本，都带有 ObjectID（哈希），使用包名 + 哈希加载；在有哈希的情况下，甚至连包名都不需要

调度器下发的配置文件中隐含了这个区分逻辑：
- 不带哈希 → 按包名加载（开发版）
- 带哈希 → 精确加载（正式版）

### 4. 混合模式

在开发版系统中也可以走正常的安装流程（从应用商店下载）。此时会出现混合情况：

- 部分组件通过包名加载（本地编译构建的）
- 部分组件通过哈希加载（从应用商店安装的）

> **遗留问题：** PackageEnv 有严格模式和非严格模式。严格模式依赖内部数据库做包名与哈希的匹配校验；非严格模式（开发模式）走目录搜索。混合使用两种模式时——即一个包在数据库中不存在但目录中存在——需要确认 PackageEnv 是否支持这种配置。过去的设计假设开发模式和正式模式不会混用，但现在实际上是混用的。

## 二、自动升级流程

### 1. 进程存活判断（判活）

NodeDaemon 本质上是**状态推演**逻辑：不断根据调度器下发的配置（目标状态）来检测目标进程是否存在。

**核心问题：** 传统的检测方式是基于包名/进程名，无法判断当前运行的进程是否是调度器配置文件中指定的那个带哈希的精确版本。

**改进方向：**

- 如果调度器下发了哈希，判断进程是否存在时必须判断其**完整路径**（PackageEnv 的实现保证目录路径中一定包含哈希）
- 或者由 NodeDaemon 维护一个列表，每次启动后写入 PID + 包名 + 包哈希的映射

> **权衡：** 维护列表的方式更准确，但排除了状态推导能力——它假设所有进程都是自己拉起的。如果是其他方式（如运维人员手动启动）启动了目标版本的服务，列表方式无法感知。而状态推演方式下，NodeDaemon 只要看到目标 ObjectID 对应的进程在运行，就不做任何操作。

### 2. 判活的效果

基于 ObjectID 判活有一个好处：**不区分新版本还是旧版本**，只要不是目标版本在运行，就认为"不存在"，从而触发启动新版本。

### 3. Lazy Install（懒安装）

启动新版本前，先判断 PackageEnv 中是否已安装该包：

1. 如果未安装，通过 **PackageID**（命名对象）从分布式对象存储（NamedStore）中获取
2. NamedStore 是系统中**性能最高的分布式存储组件**，比文件系统更轻量
3. 通过 ObjectID 打开 Reader → Unzip 到 PackageEnv 的实际目录

**关键特性：**

- **原子性安装：** 安装过程 100% 原子性，安装前已对包的哈希做过校验，永远不存在"下载到一半"或"安装到一半"的问题
- **自修复能力：** 即使目录被误删除，只要分布式对象存储还在，就能自动还原
- 这也是为什么一定要用 ObjectID 的原因——分布式系统如果依赖传统运维脚本，必然会碰到"装一半"的问题

### 4. 启动与停止脚本的设计

- 所有启动脚本的**第一行**是执行停止操作
- **停止脚本**使用通配符杀进程（泛化停止，停止该包所有版本）
- **检测脚本**使用精确路径检测

> **注意事项：** 这种"有的地方用精确控制，有的地方用泛化控制"的细节逻辑非常关键，任何一处搞错都会导致状态不一致。当前这些脚本由 NodeDaemon 依靠 PackageEnv 的路径规则拼接而成，不够显式，多次出现工程师误改破坏状态依赖的情况。**建议将这套脚本作为 PackageEnv 基础设施的一部分来升级维护。**

## 三、卸载与版本清理

### 清理旧版本的策略

对于非开发版（严格版本模式），采用**定期全量清理 + Lazy Install 恢复**的策略：

1. NodeDaemon 设置一个每日清理点
2. 在清理点将 PackageEnv 中所有解压目录全部删除
3. 通过 Lazy Install 机制，需要的包会被自动恢复

这种方式避免了设计复杂的旧版本清理逻辑。

### NodeDaemon 自身的升级

NodeDaemon 作为维护服务，自身也需要升级：

1. 挂载一个极小的辅助脚本
2. 该脚本负责：删除旧版 NodeDaemon → 通过 Lazy Install 拉取新版 → 启动
3. NodeDaemon 不是直接启动的进程，而是通过一个**永远不变的 start 脚本**启动

> 这样可以避免 PackageEnv 长期运行后出现"版本爆炸"问题（类似 Python 依赖爆炸、npm 依赖爆炸），引入 ObjectID + Lazy Install 后，通过简单的全量清理即可解决。

## 四、PackageEnv Database 的角色变化

### 过去的逻辑

- PackageEnv 内部有一个 meta 数据库，支持版本运算、latest 标签等功能
- 自动升级依赖将本地数据库与服务器源（Source）的数据库做同步
- 同步后 latest 版本改变 → 触发 Lazy Install → 拉取新版本

### 现在的变化

在当前 BuckyOS 架构下：

- 不再需要重型的本地数据库同步机制
- 每个节点只需要缓存调度器下发的那些包的 meta 信息即可
- 而这些 meta 恰好在调度器创建运行进程时就已经保存了
- 旧的数据库语义在当前环境下不应该再使用
- 甚至可以退化成纯开发模式语义，避免纠结数据库数据正确性的问题
- 清理缓存很容易，但过去从未清理过这个数据库（因为数据量小被忽略了）

> **结论：** 如果所有数据库内容都是从调度器下发的配置中零散收集的，那么整个 PackageEnv 目录基本可以回到清空状态。

## 五、关于 PackageEnv 组件的反思

### 现状

PackageEnv 是一个非常早期的核心组件，最初引入时理所当然（任何系统都需要包管理和自动升级），还做了 Python 到 Rust 的移植。

### 反思

现在来看，BuckyOS **不需要这么强力的组件**。实际需要的核心功能：

- 一个 `install` 函数（即 Lazy Install）
- 一组目录规则
- 没有数据库、没有版本运算、没有缓存之后，一切都变得比较纯粹

**纠结点：**

- 删除或简化一个运行很久的老组件是否会带来风险？
- 修改上游 PackageEnv 以适配 BuckyOS 新需求，是否会让 PackageEnv 对其他使用方不够纯粹？
- 可能一个由 NodeDaemon 管理的、更轻量级的简单目录规则组件就足够了

## 六、包更新的完整流程（发行者视角）

### 第一步：发行者发布新版本

1. 构建包，生成 ObjectID
2. 上传到自己的 **RepoServer**
3. 将 **PackageMeta**（包含最新版本信息）广播出去

### PackageMeta 的三个存储/查询位置

| 位置 | 说明 |
|------|------|
| **BNS（区块链命名服务）** | 最权威、最稳定，但更新成本较高，一般不作为首选 |
| **包索引器（Source）** | 允许使用者订阅多个源，PackageMeta 都带签名，下载时一定会做本地验证 |
| **发行者自身（OOD）** | 通过 owner 名字反推出 OOD 域名，使用标准 Repo 协议直接查询 ObjectID 或按包名列出所有发行版本 |

发行者至少需要将 PackageMeta 存入以上三个位置中的**至少一个**。

### 第二步：待升级端（OOD）获取更新

1. 通过某种方式（自动/手动）发现新版本的 PackageMeta
2. **Collect（收藏）：** 将 PackageMeta 元数据收藏到自己的 Repo
3. 通知用户发现新版本（或根据系统配置自动下载）
4. 将 PackageMeta 对应的文件下载到 Repo 的 NamedStore

> **本质：** 这是一个与包加载无关的、业务无关的通用逻辑——"同一个名字的对象，其 ObjectID 发生了更新"。

### 第三步：用户触发升级

1. 在调度器中将应用配置文件的 ObjectID 从旧版本改为新版本
2. 调度器下发新配置到 NodeDaemon
3. NodeDaemon 在目标节点上执行：
   - 判活：发现目标版本进程不存在
   - Lazy Install：从 NamedStore 中拉取并安装
   - 启动新版本进程（启动脚本第一行先停止旧版本）

### 核心容错设计

> **系统可靠性保证在调度器构造配置文件之前。** 对于底层基础设施来说，走到它们的流程时，都假设所需文件已经存在于 NamedStore 中。它们不关心文件从哪里来，只需要知道"你要我运行指定版本，这个版本所需的文件必须在 Store 里已经准备好"，然后通过 Lazy Install 完成即可。


## 七、BuckyOS的xar包

xar包与apk类似，属于“用户拿到”就能安装的包。默认只能从0到1或则升级，无法自动降级。
buckyos的安装器，可以 xar_installer $xar_path $parms 通过命令行参数来完成安装
