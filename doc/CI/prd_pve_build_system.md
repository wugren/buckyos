# pve-build-system CI/CD 与签名 PRD

## 1. 背景

`pve-build-system` 是私有自动化仓库，运行在 BuckyOS 组织控制的本地 PVE 集群和实体服务器环境中。它负责定期检查、CI 打包、签名包生成、CD 验证，以及后续 ROM 生成流程。

本 PRD 只描述 `pve-build-system` 在安装包与打包流程中的产品需求和验收标准。

## 2. 模块定位

`pve-build-system` 负责：

- 定期或手动触发 CI。
- 调度不同平台打包机器。
- 调用 BuckyOS 仓库内指定打包脚本生成未签名安装包。
- 使用商业合作伙伴授权的证书或云签名 token 生成签名安装包。
- 执行 CD 验证。
- 后续基于具体 Linux 安装包生成 ROM。

`pve-build-system` 不负责：

- 定义安装包组件清单。
- 定义文件覆盖规则。
- 实现平台安装器产品行为。
- 构建当前仓库之外的隐式组件逻辑。

安装包内容规则必须以 BuckyOS 仓库内配置和打包脚本为准。

## 3. 目标用户

- CI 系统维护者：需要持续产出各平台安装包。
- 发布负责人：需要从同一编译结果生成无签名包和签名包。
- 商业合作伙伴：授权 BuckyOS 使用签名证书或云签名 token。
- 质量验证负责人：需要用真实安装包执行 CD 验证。
- 硬件合作伙伴：后续需要可追溯到 Linux 安装包的 ROM。

## 4. CI 打包需求

### 4.1 触发方式

CI 支持：

- 定时触发。
- 通过 bot 手动触发。

### 4.2 平台覆盖

CI 需要覆盖：

- Windows 10/11 amd64，输出 exe。
- macOS >=14 amd64、aarch64，输出 pkg。
- Linux 主流发行版，Ubuntu 系优先，amd64、aarch64，输出 deb。
- Linux rpm 视可行性决定是否本期输出。

Linux rpm 规则：

- 评估同一台 Ubuntu 打包机同时生成 deb 和 rpm 的可能性。
- 如果 deb 和 rpm 可以在同一打包机稳定生成且行为一致，本期同时产出两个安装包。
- 如果不能，则本期只产出 deb，rpm 后续版本再交付。

### 4.3 产物要求

产物命名规则：

```text
buckyos-{os}-{platform}-{version}+build{builddate}
```

版本号来源：

- 从编译好的二进制调用 `--version` 得到。

元数据要求：

- 生成构建元数据。
- 不要求生成 checksum。
- 产物永久保留。

验收标准：

- CI 能在目标平台生成对应安装包。
- CI 生成的安装包来自 BuckyOS 仓库内指定打包脚本。
- CI 不绕过 BuckyOS 仓库内组件清单和覆盖规则配置。
- 构建失败时能定位失败平台和失败阶段。

## 5. 签名包生成

### 5.1 产物类型

CI 需要生成：

- 无签名 BuckyOS 安装包。
- 商业签名 BuckyOS 安装包。

无签名包面向：

- 开发人员发给测试或朋友。
- GitHub release 下载。
- CD 验证。

签名包面向：

- 商业合作伙伴运营的网站。
- 这些网站可能提供面向 BuckyOS 用户的收费服务。

### 5.2 签名流程边界

要求：

- 首先从同一批编译输出生成未签名 staging。
- 无签名包由未签名 staging 直接生成。
- 签名包从同一份未签名 staging 派生出独立签名 staging。
- 签名 staging 经过二进制签名、打包和安装包签名流程。
- 签名失败时阻断签名包发布。

验收标准：

- 无签名包和签名包都来自同一批编译输出。
- 无签名包本身不签名，且不能包含已签名二进制。
- 签名包能验证二进制签名和安装包签名是否正确。

### 5.3 平台签名要求

| 平台 | 签名要求 |
| --- | --- |
| Windows | 包内二进制和安装包本身都需要签名 |
| macOS | 包内二进制和安装包本身都需要签名 |
| Linux | 按通用 Linux 包签名方式处理 |

### 5.4 商业证书授权

商业合作伙伴负责：

- 提供签名证书或云签名 token 的使用授权。
- 将签名所需证书文件或云签名 token 提供给指定 BuckyOS 核心开发者。

BuckyOS 负责：

- 后续签名操作由 BuckyOS 开发者执行。
- 证书文件或云签名 token 保存在 BuckyOS 开发者管理的线下服务器中。
- 证书文件或云签名 token 提供给运行在 BuckyOS 组织完全控制的实体服务器上的 CI 系统使用。

## 6. CD 验证需求

### 6.1 输入与环境

CD 输入：

- CI 产出的无签名安装包。

CD 覆盖安装包类型：

- exe。
- pkg。
- deb。

CD 覆盖平台：

- Windows。
- macOS。
- Ubuntu。

rpm 验证要求：

- 如果本期产出 rpm，先乐观估计 deb 和 rpm 行为一致。
- CD 只验证 deb，不要求覆盖 rpm。

CD 虚拟机环境：

- 干净系统。
- 已提前安装好所有不能自动安装的必要依赖。
- 需要网络环境。

### 6.2 固定验证流程

CD 固定流程：

1. 在干净系统中执行全新静默安装。
2. 安装完成后执行激活流程。
3. 访问 BuckyOS control panel。
4. 能访问 control panel 即判定激活成功。

安装成功判定：

- Windows/macOS 安装后 BuckyOSApp 程序存在。
- Windows/macOS 安装后 buckycli 可调用。
- Linux 安装后 buckycli 可调用。
- Linux 不要求 BuckyOSApp 存在。
- buckyos service 已启动。
- service 启动后进入待激活状态还是已激活工作状态，由 buckyos service 自身决定。

本期不进入 CD 的场景：

- 覆盖安装。
- 依赖缺失。
- rpm 安装。
- 签名包安装。

失败处理：

- CD 失败时保留现场。
- CD 失败时保留相关日志和产物。

发布阻断标准：

- 静默安装失败阻断发布。
- 安装后激活失败阻断发布。
- 激活后基础功能验证失败阻断发布。

## 7. ROM 生成需求

本期不交付 ROM 实现，但 `pve-build-system` 预计承载后续 ROM 生成流程。

本期只记录 ROM 需求边界：

- ROM 基于 Ubuntu 或 Debian，具体发行版待定。
- ROM 运行在类似小型 PC 的机器上。
- CPU 架构为 amd64。
- ROM 由某个具体 Linux 安装包生成。
- ROM 必须能追溯到用于生成它的具体 Linux 安装包。
- ROM 开机后启动 buckyos service。
- service 启动后进入待激活状态还是已激活工作状态，由 buckyos service 自身决定。

非目标：

- 本期不定义 ROM 激活入口。
- 本期不定义 ROM 网络异常行为。
- 本期不定义 ROM 升级和维护策略。
- 本期不要求 ROM 进入 CD 验证。

## 8. 与其他模块的关系

- `buckyos-devkit` 提供构建和安装到 target rootfs 的基础能力。
- BuckyOS 仓库打包脚本定义安装包内容、覆盖规则和平台未签名安装包生成。
- `pve-build-system` 只调用 BuckyOS 仓库内指定打包脚本产生安装包，不重新定义安装包内容。
- CD 必须消费 CI 产出的安装包，不应绕过安装包直接部署源码或 staging。
- ROM 后续必须消费某个具体 Linux 安装包，不应绕过 Linux 安装包直接从源码生成。

## 9. 验收标准汇总

- 能定时或手动触发多平台 CI。
- 能调用 BuckyOS 仓库打包脚本生成未签名安装包。
- 能从同一批编译输出生成无签名包和签名包。
- 无签名包不包含已签名二进制。
- 签名包能验证二进制签名和安装包签名。
- CD 使用无签名安装包完成静默安装、激活和 control panel 访问验证。
- CD 不覆盖 rpm、签名包、覆盖安装和依赖缺失场景。
- 后续 ROM 生成必须可追溯到具体 Linux 安装包。
